# ARK LLM 公众号日报改写设计

**目标**

在现有 AI 每日新闻速递流水线中接入火山 ARK LLM，把程序筛选后的结构化候选素材改写成一篇适合公众号草稿箱发布的中文成稿。文章风格不是资讯搬运，而是“今天我筛了几条最值得跟的”，重点是判断和取舍。

**范围**

- 保留现有抓取、去重、排序、公众号草稿发布链路
- 新增 ARK 配置读取和 LLM 写稿模块
- 修正 GitHub Trending 解析，确保能稳定提取真实仓库
- 调整栏目选择逻辑，避免“今日重点”和后续栏目重复
- 调整 CLI 输出，发布时默认只输出结果，不把整篇正文回显到终端

不在本次范围内：

- 自动正式群发
- 多模型回退
- 长期记忆、向量检索或 embedding 排序
- 图文封面设计风格升级以外的公众号运营能力

## 现状问题

当前版本的主要问题有四类：

1. `arXiv` 内容直接使用英文摘要，长度失控，阅读体验像原始数据，不像公众号文章。
2. `DigestComposer` 直接按全局得分取前五条做“今日重点”，后续栏目继续复用同一批条目，内容重复。
3. GitHub Trending 抓取器当前匹配第一个链接，容易把非仓库链接误识别为项目链接。
4. 正文生成是规则模板，输出形态是“摘要/价值”列表，不具备个人开发者筛选后的判断感。

## 推荐方案

采用“程序控结构，LLM 写成稿”的混合方案。

- 程序负责：
  - 抓取
  - 去重
  - 排序
  - 分类
  - 栏目选题
  - 公众号发布
- LLM 负责：
  - 写标题
  - 写导语
  - 把每个栏目中的素材改写成中文成稿
  - 输出结尾一句判断

这样做的原因很直接：

- 结构选择仍由程序控制，避免整篇内容跑偏
- LLM 只处理中文表达和判断性改写，稳定性更高
- 文章风格可以升级，但发布链路和测试边界仍然可控

## 目标文风

文风固定为“个人开发者筛选日报”，具体约束如下：

- 使用中文
- 第一人称，但不过度表演
- 面向开发者和技术从业者
- 明确说出“为什么值得跟”
- 允许轻量判断，不写营销腔
- 不出现“摘要：”“价值：”这种机器标签
- 不直接粘贴英文长摘要
- 每条内容控制在 2 到 4 句

参考语气：

- “今天我筛了几条值得跟的 AI 动态，偏技术和工具侧。”
- “这条我会优先看，因为它已经不只是概念，而是开始影响开发流程。”

## 架构调整

### 1. 来源抓取层

保留现有 RSS 和 GitHub 抓取器，但修正 GitHub Trending 解析规则。

GitHub 抓取器的新规则：

- 只接受路径形如 `/<owner>/<repo>` 的链接
- 过滤 `login`、`features`、`topics`、`collections` 等非仓库路径
- 优先从标题区域提取仓库名，而不是抓取 article 中第一个链接
- 继续提取描述和 stars today

目标是确保 GitHub 栏目稳定拿到 2 到 5 条可用项目。

### 2. 规范化与排序层

保留现有 `DigestItem` 模型和 `ItemRanker`，但调整栏目选题逻辑。

新增一个“栏目选择器”阶段，位于排序之后、LLM 写稿之前：

- `今日重点`：从全量候选中精选 3 到 5 条，强调跨来源混排
- `GitHub 新项目 / 热项目`：单独保留 2 到 3 条 GitHub 项目
- `AI 技术进展`：保留 2 到 3 条新闻或论文更新

约束：

- 任一条目只能出现在一个栏目里
- GitHub 栏目设置最低保底，只要有足够合格候选，就不会被 arXiv 完全挤掉
- 同一来源、同一 dedupe key 仍然遵守 7 天内不重复出现的规则

### 3. LLM 写稿层

新增 `ARKArticleWriter`，输入为程序整理后的结构化素材，输出为整篇公众号正文。

输入内容只包含必要字段：

- 标题
- 来源
- 链接
- 分类
- 发表时间
- 原始摘要或描述的截断版本
- 排序分数
- GitHub stars 增长等结构化信息

LLM 不直接访问网页，也不直接参与抓取排序。这样可以减少上下文噪声，也便于测试。

### 4. 公众号发布层

保留现有微信草稿发布器和自动封面生成逻辑，只替换正文来源：

- 旧路径：规则模板直接生成 Markdown
- 新路径：结构化素材 -> LLM 生成 Markdown -> 转公众号 HTML -> 提交草稿

## 数据流

完整数据流如下：

1. `collector.collect()` 获取原始条目
2. `deduper.filter()` 做 7 天窗口去重
3. `ranker.rank()` 计算排序分数
4. `section_picker.pick()` 生成三个不重复栏目
5. `payload_builder.build_article_input()` 生成 LLM 输入载荷
6. `ARKArticleWriter.write()` 生成整篇 Markdown
7. `publisher.publish()` 提交公众号草稿

失败降级规则：

- 单个抓取源失败，不中断全流程
- LLM 调用失败时，整次发布标记为失败，不降级回旧的英文模板
- 栏目候选不足时，整次发布标记为 `skipped`

## Prompt 设计

Prompt 采用系统提示词 + 结构化 JSON 输入。

系统约束要点：

- 你在写一篇面向开发者的 AI 日报
- 你的身份是每天筛选信息的人，不是媒体编辑部
- 你必须有判断和取舍，说明为什么值得跟
- 所有输出必须是中文
- 不允许照抄输入中的英文摘要
- 不允许编造不存在的事实、数字和链接
- 每一条都要基于输入素材写，不得扩写到输入之外

输出格式要求：

- 使用 Markdown
- 包含一个标题
- 包含一个导语段
- 包含三个二级栏目：
  - `## 今日重点`
  - `## GitHub 新项目 / 热项目`
  - `## AI 技术进展`
- 每个栏目下的每条内容使用短小段落，而不是“摘要/价值”子弹标签
- 结尾附一段一句话总结

为减少跑偏，Prompt 中显式禁止：

- 空泛套话
- 伪深度判断
- 夸张词和营销词
- 重复同一条素材

## 配置设计

在现有 `.env` 基础上新增 ARK 相关配置：

- `ARK_API_KEY`
- `ARK_BASE_URL`
- `ARK_MODEL`
- `ARK_TIMEOUT_SECONDS`，默认 30

当前用户已经提供了 `ARK_EMBEDDING_MODEL`，但本期不接入主流程，暂不作为必填配置。

当 ARK 配置缺失时：

- 允许 `--dry-run` 继续使用旧规则稿，仅用于调试抓取链路
- `--publish` 直接报错，避免把旧风格英文稿继续发到公众号

## 测试策略

新增或调整测试覆盖以下行为：

1. GitHub 抓取
   - 只解析真实仓库链接
   - 不会把 `Star` 或 `login` 链接识别为项目

2. 栏目选择
   - `今日重点`、`GitHub`、`AI 技术进展` 三个栏目互不重复
   - GitHub 有合格候选时能够入选

3. LLM 输入拼装
   - 输入字段完整
   - 摘要会截断
   - Markdown 指令包含栏目和文风约束

4. LLM 客户端
   - 能正确向 ARK 发请求
   - 超时与非 200 响应能正确抛错

5. Pipeline 行为
   - `--publish` 且缺少 ARK 配置时失败
   - LLM 成功时返回 `published` 或 `composed`
   - LLM 失败时不继续提交公众号草稿

6. CLI 行为
   - 发布成功时不回显整篇正文
   - `--output` 时仍然可以落盘正文

## 风险与约束

主要风险有三类：

1. LLM 可能生成过度修饰的文字。
   - 处理方式：严格 Prompt 约束，并对输出执行基础风格校验，拦截明显营销腔和重复段落。

2. LLM 可能编造未出现在输入中的结论。
   - 处理方式：输入只给结构化素材，Prompt 明确禁止补充未提供事实。

3. 发布链路对成稿格式更敏感。
   - 处理方式：保留现有 Markdown -> 微信 HTML 的转换路径，不直接让 LLM 输出 HTML。

## 成功标准

完成后，成功标准如下：

- 公众号草稿全文为中文
- `arXiv` 条目不再出现整段英文摘要
- `今日重点` 与后续栏目不重复
- GitHub 栏目在有候选时能稳定出现
- 正文整体更像个人开发者筛选后的公众号文章
- 发布成功时终端输出简洁，不再打印整篇正文

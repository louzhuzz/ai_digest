# 短版专业简报成稿方案

## 目标

把当前“热点汇总式”成稿，调整为更像开发者专业简报的短版文章。

目标效果：
- 全文控制在 800 到 1200 字
- 读者 5 分钟内能看完
- 开头先给判断，不先堆素材
- 重点更少，但每条都更有“这意味着什么”
- 不再追求栏目完整，改为追求判断密度和阅读效率

本方案只针对“成稿风格与生成机制”优化，不扩展到采集、发布或界面改造。

## 当前问题

当前链路已经能抓取、排序、聚类、生成并发布，但成稿层仍有几个明显问题：

- 文章容易像“多条热点的顺序拼接”，而不是一篇完整简报
- 虽然已经加入 LLM 重写，但模型仍可能平均分配篇幅，导致主次不够分明
- 标题和导语有时过于通用，缺少“今天到底发生了什么”的中心判断
- 正文容易回到“逐条总结”模式，像摘要集合，不像编辑判断
- 栏目结构过于完整时，会稀释短版内容应有的节奏感

这些问题不会通过继续增加 Markdown 样式直接解决，根因在于：
- writer 输入还不够明确地区分“主重点”和“次重点”
- prompt 仍允许模型过度自由地铺开内容
- 成稿目标没有被收紧到“短版专业简报”这一单一风格

## 设计原则

本次优化遵循以下原则：

1. 先结论，后展开
2. 只保留少量真正值得写的重点
3. 每一段都要服务于“今天该跟什么”这个问题
4. 语气稳，不口语化，不营销化
5. 写成一篇完整简报，而不是栏目模板填空

## 推荐成稿结构

### 1. 标题

要求：
- 标题直接点出当天最值得跟的一条变化
- 不再默认使用“AI 每日新闻速递”之类通用前缀
- 标题长度控制在 12 到 24 字，偏消息判断型

示例方向：
- OpenAI 再推新能力，AI 开发工具链开始重排
- 今天最值得跟的，不是模型，而是 AI 编码工作流在成形
- GitHub AI 项目继续升温，智能体开发开始进入工程化阶段

标题不是情绪化标题党，而是“带判断的信息压缩”。

### 2. 开场总判断

位置：标题后第一段。

要求：
- 2 到 3 句
- 先给当天整体判断，不先罗列素材
- 回答“今天的主线是什么”

示例作用：
- 说明今天热点偏模型发布、偏产品更新，还是偏开源项目爆发
- 给读者一个阅读框架，后文都围绕这个框架展开

### 3. 主重点区

章节建议：`## 今天最值得跟的两件事`

要求：
- 只保留 2 条主重点
- 每条控制在 120 到 180 字
- 每条都包含三层信息：
  - 发生了什么
  - 为什么这条比别的更值得跟
  - 它对开发者意味着什么

这里不使用“摘要：”“价值：”“为什么值得跟：”等标签，而是写成自然段。

### 4. 次重点区

章节建议：`## 其余还值得看什么`

要求：
- 只放 2 到 3 条次重点
- 每条控制在 60 到 90 字
- 重点写“值得继续关注的原因”，不展开背景
- 用来补充读者不应漏掉的次级动态

### 5. 收尾句

要求：
- 1 句即可
- 总结今天的主线，不再新开信息点
- 用于收束节奏，而不是重复前文

## 内容选择规则

### 主重点选择

默认从已有排序结果中选出 2 条主重点，但不是简单取前 2 名。

优先规则：
- 优先能代表“今日主线”的事件
- 优先对开发者有直接影响的更新
- 优先一手来源、多源重复验证、信号强的内容
- 如果新闻显著强于项目，主重点可为 2 条新闻
- 如果当天项目明显更强，可为 1 条新闻 + 1 条项目，或 2 条项目

### 次重点选择

从剩余高分项里补 2 到 3 条。

要求：
- 不与主重点重复
- 不写成主重点的补充材料
- 更像“你可以顺手记下的后续动向”

### 不入选内容

以下内容应更积极地被压缩或舍弃：
- 只是泛泛而谈的资讯汇总
- 缺少开发者相关性的行业动态
- 只是论文摘要但对当天圈内讨论不构成主线的内容
- 信息重复但没有新增含义的二次报道

## 生成链路调整

### 现状

当前项目已经具备：
- 候选池抓取
- 去重、排序、来源配额
- 聚类与 cluster tag
- writer 成稿

但“文章怎么组织”仍主要交给 writer 自行消化。

### 调整后

建议把成稿前的输入再前移一步，明确给 writer 一个更窄的写作任务：

1. 从 quota 后候选项中选出：
   - `lead_items`: 2 条
   - `secondary_items`: 2 到 3 条
2. 生成明确的 article_input：
   - 今日主线
   - 主重点列表
   - 次重点列表
   - 明确禁止平均分配篇幅
3. writer 按固定“短版专业简报”风格输出

这意味着：
- composition 不再承担成稿模板职责
- payload builder 不只是“把 items 塞进 JSON”，而是要生成简报语义层输入
- linter 继续只做发布底线检查，不评价审美

## Prompt 约束建议

writer prompt 应明确增加以下要求：

- 你写的是“开发者专业简报”，不是公众号软文
- 全文目标 800 到 1200 字
- 必须先给总判断
- 只展开 2 个主重点
- 其余信息只做 2 到 3 条短条补充
- 不要平均照顾所有候选项
- 不要写“今天没有重大新闻”这种空转句
- 不要把英文摘要改写成长段中文说明
- 多写判断，少写背景铺垫
- 语气平稳、专业、克制

## 数据流影响

本设计只改成稿层，不改采集和发布层。

受影响模块：
- `section_picker` 或其后继选择器：需要产出 lead / secondary 结构
- `payload builder`：需要生成更强约束的 article input
- `llm_writer`：需要切到短版专业简报 prompt
- `tests`：需要增加对篇幅、主次结构和开场总判断的行为测试

不直接改动：
- collectors
- dedupe
- wechat publisher
- webapp 页面结构

## 验收标准

当该方案完成后，生成结果应满足：

1. 标题不再是通用日报标题，而是当天一条具体变化的判断性标题
2. 首段能在 2 到 3 句内说明“今天主线是什么”
3. 正文只保留 2 个主重点和 2 到 3 个次重点
4. 全文整体更像一篇短简报，而不是栏目堆叠
5. 阅读时能在前 20 秒内知道今天最该跟什么
6. 在热点池一般的日子里，也不会写成空泛套话

## 风险与边界

### 风险

- 若当天候选池质量不足，强行输出“判断感”可能导致文章显得牵强
- 若主重点选择规则过硬，可能误伤一些需要灵活处理的日子
- 若字数限制过死，可能导致重要背景被压缩过度

### 边界处理

- 如果当天只有 1 条足够强的主重点，允许第 2 条主重点降为较短段落
- 如果项目明显强于新闻，允许整篇主线偏项目，但仍保持“专业简报”语气
- 如果候选池不足，仍应自然降级，而不是硬凑成完整分析文

## 实施建议

建议先做最小闭环：

1. 在成稿前新增“主重点 / 次重点”选择层
2. 收紧 prompt，改成短版专业简报风格
3. 保持现有发布链路不变
4. 先用 dry-run 验证 3 到 5 天样本输出是否更稳

如果效果明显，再决定是否进一步把 outline 两阶段链路也改成“先生成简报结构，再成稿”的专用版本。
